Small Talk with Claude — Improvement Plan

Review of the self-hosted blog stack at edwardhong.net: the mdpub publishing CLI, the blog-api interactivity service (comments / reactions / contact), and the Zola site itself. The codebase is in good shape overall — the review surfaced 1 critical data-loss risk, 2 high-priority bugs, and a set of medium/low hardening and polish items, organized into four phases below.

What's already solid

Worth naming, so the plan doesn't churn what works.

  • Real test discipline: 76 mdpub tests, route-level blog-api tests, external commands mocked behind a Runner trait.
  • Security fundamentals done right: constant-time admin-token comparison, X-Forwarded-For only trusted from loopback (spoof-aware), honeypot fields, Caddy request-body cap, comments moderated by default.
  • Frontend XSS discipline — user content rendered exclusively via textContent, with the rule documented at the top of interact.js.
  • Coherent, hand-built site identity with dark/light theming, lazy-loaded card media, reduced-motion support, and an atom feed.
  • Good docs: tutorial, server setup, and a deploy-debugging postmortem.

Findings

Ordered by severity. Each carries a concrete failure scenario and a proposed fix.

criticalF1 · data loss

Repo has drifted from the live site — the next publish will delete two live posts

.mdpub-state.json tracks 6 published articles, but only 4 imported content directories exist in blog/content/blog/ (on this branch, on main, and on origin/main). The Day 5 (ClickHouse MergeTree) and Day 6 (embedding table) imports were never committed. Because mdpub publish runs zola build over whatever is in blog/content and then rsync --delete, publishing anything from a fresh checkout will silently remove both live posts from edwardhong.net.

.mdpub-state.json · mdpub/src/deploy.rs (rsync --delete) · blog/content/blog/

Fix: re-run mdpub publish for Day5/ClickHouse_MergeTree.md and Day6/from-text-to-vectors.md to regenerate the imports, commit them, and merge to main. Then add a drift guard to mdpub: publish refuses (and status warns) when the state file tracks a slug with no matching directory under blog/content/blog/.
highF2 · bug

Anonymous comments are rejected, but the UI promises they work

The comment form labels the name field "(optional)" and the renderer falls back to "anonymous" — but create_comment validates author with validate_len("author", &payload.author, 1, 80), so an empty author returns 400. Readers who skip the name field get a generic "could not submit comment" error and their comment is lost.

blog-api/src/routes/comments.rs:68 · blog/templates/page.html:39 · blog/static/js/interact.js

Fix: allow empty author server-side (min 0, or default to "anonymous" on insert), add a regression test for the empty-author case, and surface the API's error message in the form status instead of a generic failure.
highF3 · robustness

Handler .unwrap()s can poison the shared DB mutex and kill the service

Every route locks Arc<Mutex<Connection>> with .unwrap() and calls prepare/execute/query_map(...).unwrap(). One SQL error (disk full, locked DB, malformed data) panics the handler task — and because the panic happens while holding the mutex, the mutex is poisoned and every subsequent request panics until the process is restarted.

blog-api/src/routes/{comments,reactions,admin,contact}.rs

Fix: introduce an AppError → 500 conversion (axum IntoResponse), replace handler unwraps with ?, and recover poisoned locks via lock().unwrap_or_else(|e| e.into_inner()). Add a test that a failing statement yields 500, not a hang.
mediumF4 · process

No CI — and formatting has already drifted in both crates

There is no .github/workflows/; the 76-test suite only runs when someone remembers. cargo fmt --check currently fails in both mdpub and blog-api — exactly the drift CI would have caught.

repo root (no .github/) · mdpub/src/cli.rs · blog-api/src/config.rs

Fix: one workflow running cargo fmt --check, cargo clippy -D warnings, and cargo test per crate on push/PR; run cargo fmt once to clear the existing drift.
mediumF5 · durability

No backup story for the comments/reactions database

All reader data lives in a single SQLite file on the $5 Linode. A disk failure or a bad deploy loses every comment, reaction, and contact message permanently.

blog-api (SQLite at BLOG_API_DB_PATH) · docs/server-setup.md

Fix: nightly sqlite3 .backup via cron with 7-day rotation, synced off-box (even rsync back to the Mac works). Litestream is the fancier option if you want continuous replication.
mediumF6 · seo

No Open Graph / Twitter cards, and canonical_url is silently dropped

Article pages emit no og:* or twitter:* meta, so links shared to social/Slack/Discord render bare. mdpub faithfully carries canonical_url frontmatter through import — but no template renders a <link rel="canonical">, so cross-posted articles get no canonical signal at all.

blog/templates/base.html (head) · blog/templates/page.html · mdpub/src/frontmatter.rs:128

Fix: add og:title/description/url/image (first page asset works as the image), twitter:card, and canonical link to the head block. ~20 lines of Tera.
mediumF7 · resource leak

Rate limiter never evicts expired windows

The fixed-window map is keyed by (ip, route) and entries are never removed, so memory grows monotonically with every unique client IP for the life of the process. Slow on a small blog, but a scraper sweep or bot wave inflates it — and it never comes back down.

blog-api/src/ratelimit.rs:72

Fix: sweep expired entries opportunistically inside check() (e.g. every N calls) or via a background task; add a test that stale keys are dropped.
lowF8 · hardening

No security headers on the static site

The Caddy snippet proxies /api/* but sets no CSP, X-Content-Type-Options, or Referrer-Policy. The site's JS surface is small and first-party, which makes a strict CSP cheap to adopt now (the Google Fonts origin is the only third party — see F9).

blog-api/deploy/caddy-snippet.conf · /etc/caddy/Caddyfile

Fix: add a header block: CSP allowing self + fonts origins, nosniff, strict-origin-when-cross-origin.
lowF9 · performance

Multi-megabyte GIFs and third-party fonts weigh down page loads

Post diagrams ship as GIFs up to 4 MB (rebac.gif) — the RBAC post carries ~6.7 MB of GIF alone. Fonts load from Google Fonts, adding a third-party dependency, a render-blocking request chain, and an EU-privacy wrinkle.

blog/content/blog/*/ (gifs) · blog/templates/base.html:19–21

Fix: convert animated diagrams to looping <video> (MP4/WebM is typically 8–15× smaller; could be an mdpub image-pipeline step), and self-host the three font families under blog/static/fonts/.
lowF10 · workflow

Pending comments and contact messages are invisible until you poll

Moderation requires remembering to run curl with the admin token; a comment can sit pending for weeks unnoticed. Same for contact messages.

README.md (moderation section) · approve_comments.sh

Fix: a daily server cron that checks pending counts and pings you (ntfy.sh / email via msmtp) only when nonzero. Alternatively a mdpub comments subcommand to list/approve interactively from the repo.
lowF11 · hygiene

Small repo/script hygiene items

publish.sh has no shebang, no set -euo pipefail, and an unquoted $path (breaks on filenames with spaces). Editor/agent state is committed (plans/ scratch files, .claude/, .serena/ partially tracked). Article sources live in Day1..Day6/, which won't scale past the daily-cadence phase.

publish.sh · .gitignore · repo root

Fix: harden or delete publish.sh (it's a 1-line alias), extend .gitignore, and consider articles/<slug>/ naming going forward.

Roadmap

Four phases, ordered by risk. Phase 0 is genuinely urgent — it's the only item where inaction can destroy live data.

PHASE 0

Stop the data-loss window

~1–2 h · do before any publish
  • Republish Day 5 & Day 6 to regenerate their blog/content imports; verify against the live site; commit & merge. F1
  • Add the drift guard to mdpub publish / mdpub status, with tests. F1
  • Fix the anonymous-comment 400 + regression test. F2
PHASE 1

Make correctness automatic

~half a day
  • GitHub Actions: fmt + clippy + test for both crates; run cargo fmt once to clear existing drift. F4
  • Replace handler unwraps with an AppError type; un-poison mutex recovery; 500-path test. F3
  • Nightly SQLite backup cron with rotation. F5
PHASE 2

Reach & hardening

~half a day
  • Open Graph / Twitter meta + canonical link in templates. F6
  • Security headers in the Caddy config. F8
  • Rate-limiter eviction + test. F7
PHASE 3

Polish & workflow

opportunistic
  • GIF → looping video pipeline; self-host fonts. F9
  • Pending-comment notification (cron + ntfy) or mdpub comments subcommand. F10
  • Script/repo hygiene; articles/ layout for future posts. F11
  • Optional: enable Zola's search index once post count justifies it; aria-labels on reaction buttons.

Method. Reviewed all Rust sources in mdpub/ and blog-api/, Zola templates and JS in blog/, deploy scripts, docs, git history across branches, and .mdpub-state.json. Ran cargo fmt --check on both crates (both fail); test suites were not executed as part of this review. The F1 drift was confirmed against main and origin/main trees, not just the working branch.
